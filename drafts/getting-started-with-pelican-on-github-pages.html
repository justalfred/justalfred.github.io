Getting Started with Pelican on Github Pages

Date (Tue) 2015-06-30 Tags Code

I want more people to blog. I want more people to give voice to their experiences. Especially those people who think they have nothing to say. It’s not that I think we need more noise on the internet. It’s that the conversations out there are often dominated by a few loud voices that are often not representative of the masses. It’s that sometimes the most helpful thing for you to hear is that someone out there has gone through what you’re going through and can tell you something about the experience. I want more people to blog.

There are various guides out there that show how easy it is to get started publishing with Pelican. My problem is that I don’t like uncertainty, and these guides typically explain ‘what’, but not ‘why’. What are all these auto-generated files for? How do I customize the look and organization? How can I be sure I don’t expose more information than I want? Because of my insecurity, it took me a lot longer to get started than I wanted. Here is my summary of the steps I ended up taking, and an explanation of what each step does. Hopefully this will save the reader time bouncing around various tutorials, blog posts, and documentation.

Intent

I wanted a blogging platform which allows me to draft in Markdown. I didn’t want to pay much money, and I wanted to avoid having to serve a website. After a few months of indecision, I settled on using Pelican to generate a static website which I would host on Github Pages. Pelican has the advantage of being well supported and in a language that I like (Python). Github Pages is free and allows me to use a custom domain. I set everything up at the start of January, and haven’t touched it since. So this is my best recollection of how I got started.

Pelican

If you’re not familiar with the terminology so far, Pelican is a Python library that lets you generate static websites from templates. By static, we mean that the content served is the same for everyone because it’s generated in advance. Blogs are typically static while ecommerce sites (where you login, have an account, order history, etc.) are typically not. Templates save the blogger from having to muck about with verbose HTML. Markdown and reStructuredText are two templating formats that Pelican handles well. As an example, Markdown lets you write: [a link](http://blog.justalfred.com) instead of: <a href="https://blog.justalfred.com">a link</a>. Pelican itself is highly customizable and extensible. You’ll want to be somewhat familiar with Python and git to have an easy time following along. I’ve been working in Arch Linux, but these instructions will probably work in OSX as well. Windows users will have to interpret the commands and paths accordingly.

Install

So let’s get started. You can create a new python virtual environment if you want. If you’re sloppy and lazy like me, just go ahead and install a few packages.

$ pip install pelican Markdown typogrify

Markdown is, of course, only necessary if you want to use Markdown. Typogrify makes things a little prettier. Also optional.

Generate the initial boilerplate

The first thing to do now is to use the quickstart script to generate a skeleton for your blog. This step doesn’t make your blog, but it makes the files you’ll use later to make your blog. If you mess up, you can delete the generated folder and just start right over. For most of the prompts, you can just accept the default. For most of the rest, you can always change those settings later if you change your mind. In particular, though, I strongly recommend generating the Makefile.

$ pelican-quickstart


Welcome to pelican-quickstart v3.5.0.

This script will help you create a new Pelican-based website.

Please answer the following questions so this script can generate the files
needed by Pelican.


> Where do you want to create your new web site? [.] 
> What will be the title of this web site? Just Alfred
> Who will be the author of this web site? Just Alfred
> What will be the default language of this web site? [en] 
> Do you want to specify a URL prefix? e.g., http://example.com   (Y/n) 
> What is your URL prefix? (see above example; no trailing slash) http://blog.justalfred.com 
> Do you want to enable article pagination? (Y/n) 
> How many articles per page do you want? [10] 
> Do you want to generate a Fabfile/Makefile to automate generation and publishing? (Y/n) 
> Do you want an auto-reload & simpleHTTP script to assist with theme and site development? (Y/n) 
> Do you want to upload your website using FTP? (y/N) 
> Do you want to upload your website using SSH? (y/N) 
> Do you want to upload your website using Dropbox? (y/N) 
> Do you want to upload your website using S3? (y/N) 
> Do you want to upload your website using Rackspace Cloud Files? (y/N) 
> Do you want to upload your website using GitHub Pages? (y/N) y
> Is this your personal page (username.github.io)? (y/N) y
Done. Your new project is available at $SOME_DIR

At this point, you can navigate to the folder where Pelican guides you ($SOME_DIR) and find the following new files and folders:

  • develop_server.sh - This is a very useful shell script for seeing the effects of your updates and is called by the Makefile.
  • Makefile - This file defines make commands that perform the most important tasks
  • pelicanconf.py - This file contains settings to customize your site
  • publishconf.py - This file contains settings that only used when you’re ready to publish to the web
  • content/ - This folder is where you’ll put the templates and files that will be translated into the content of your site
  • output/ - This folder might not exist until you convert your content into html
  • fabfile.py - This has something to do with fabric which I don’t use

Some of these are discussed in further detail later.

Initial content

You’ll want to add some content now. There are two major types of content to distinguish: articles and pages. Articles are the main content of your site, like blog posts. Pages are relatively static, like ‘about’ or ‘contact’ pages. Let’s start with one of each.

Create a folder, content/pages/, make a new file called, about.md, and give it the following contents:

Title: About
Author: My Name

Bonjour, wilkommen a kono blog.

You can (should) replace all of that with the relevant information that you would like to have displayed.

Next, create a file called, content/test.md, and give it the following contents:

Title: Test Post
Date: 1997-08-29 02:14
Tags: testtag
Author: My Name
Summary: This is an article

Geometric growth is the discrete version of exponential growth.

Notice how the files begin with metadata that are not Markdown syntax. Pelican will interpret these fields when it translates the Markdown into HTML. You can read more about this section here.

You now have a few different ways of generating the html of your shiny new website. The most direct is to run pelican path/to/content/. This will generate a folder called, output/, which contains the static html for your blog. Equivalently, you can run make html with the same result. There are a few settings at the top of Makefile that you may wish to edit, but the defaults are probably good. What you’re asking Pelican to do here is to examine the files in content/ and the settings in pelicanconf.py to translate everything into your new website. Eventually, you’ll send the contents of output to wherever the files will be hosted—GitHub Pages, in my case.

Once you’ve done this, to view this in a web browser, you’ll need to start a webserver from within the output folder. Fortunately, Python comes with one. If you’re running python 2 (you probably are) you can run python -m SimpleHTTPServer. If you’re running python 3, the command is python3 -m http.server. Possibly easier, you can run make serve from the root folder of your project (i.e. one level above content/). The Makefile specifies where the output goes, so it will know where to go to serve.

Now open a web browser, navigate to http://localhost:8000 and see the results of your work so far!

Of course, the authors of Pelican have provided even more convenience for you. Given this workflow so far, every time you update a file, you have to regenerate the html to see the updates in your web browser. Stop webserver, save, make html, make serve after every change. Instead, you can run make devserver once which does three things. First, it generates the html of your site. Second, it starts a webserver at port 8000 as before. Third, it detects when content gets updated and will regenerate the html automatically. All you have to do is save content then refresh your browser to see the latest. When you’re done editing and want to stop the webserver process, run make stopserver.

Version control

It’s not necessary, but it’s probably a good idea to keep a version history of your blog contents. I’m going to assume the reader is familiar with git, github, and creating new repositories. I advise putting cache/, output/, and *.pyc in .gitignore at minimum, but the other files are probably worth tracking.. My blog contents are here. If this part isn’t clear to you, feel free to email or tweet at me, and I’ll add more detail.

Custom domain

If you own a domain that you’d like to use, this is how; otherwise skip this section. First, create a file called CNAME in a new folder, /content/extra. Edit the file so the only contents are the domain you want to use, e.g.: blog.justalfred.com. Next, edit your pelicanconf.py to add the following lines:

STATIC_PATHS = ['extra/CNAME']
EXTRA_PATH_METADATA = {'extra/CNAME': {'path': 'CNAME'},}

Now, all links generated will use your CNAME as the base domain! You can regenerate your site and see where the CNAME file ends up. Of course, there’s nothing special about the folder, extra/, but this seems to be the convention.

robots.txt

If you care about web crawlers indexing certain portions of your site, you can create a file, extra/robots.txt. This establishes a policy that web crawlers are recommended to follow. The major ones will respect your settings. Mine has the following contents:

User-agent: *
Disallow: /drafts/
Disallow: /theme/

If you didn’t follow the last portion, you will have to add the following to your pelicanconf.py:

STATIC_PATHS = ['extra/robots.txt']
EXTRA_PATH_METADATA = {'extra/robots.txt': {'path': 'robots.txt'},}

If you did, edit the relevant lines to:

STATIC_PATHS = ['extra/CNAME', 'extra/robots.txt']
EXTRA_PATH_METADATA = {'extra/CNAME': {'path': 'CNAME'},
                       'extra/robots.txt': {'path': 'robots.txt'},
               }

Typogrify

If you chose to install typogrify earlier, you’ll want to enable it. Add the following to your pelicanconf.py:

TYPOGRIFY = True

Pelican themes

If you want to customize the look of your site, you can get pretty far by using one of the packaged themes. I chose the ‘pelican-bootstrap3‘ theme because of its completeness and simplicity. It has a minimalist aeshthetic, it’s responsive, it recognizes a host of extra settings. It can be further customized by editing a custom CSS file as described here, but I haven’t gotten around to that. Whichever theme you choose, clone the ‘pelican-themes’ repo, and add the following to your pelicanconf.py:

THEME = 'path/to/pelican-themes/chosen-theme'

Time zone

You will probably want to set the time zone that your site uses. This seems to be mostly important if you set up RSS or Atom feeds later on (which I recommend; I love RSS). Simply add the following to your pelicanconf.py with the appropriate time zone:

TIMEZONE = 'America/New_York'

publishconf.py

As mentioned earlier, publishconf.py is much like pelicanconf.py, but the settings listed there are only applied for final conversion. For instance, you probably don’t want to include Google Analytics tracking in your draft versions of your site; only in the version that you finally publish.

Speaking of it, the first thing you’ll want to add is probably Google Analytics tracking. Add the tracking id to publishconf.py:

GOOGLE_ANALYTICS = "UA-XXXXXXXX-X"

If you want to generate an RSS feed (some of us still use RSS) you can do that quite simply here:

FEED_DOMAIN = SITEURL
FEED_ALL_RSS = 'feeds/all.rss.xml'
CATEGORY_FEED_RSS = 'feeds/%s.rss.xml'

If you want to use Disqus to enable comments (I haven’t decided to do that yet. Email or tweet at me if I should.), you can add the sitename in this file as well.

Once you’re ready to share your site with the world, you can generate a final version of your site. pelican content -s publishconf.py and make publish are equivalent commands to do this. It’s a good idea to delete the output folder first. Everything will be regenerated, but if you made changes that left behind some old versions of old files, you want to be sure to clear those. Note: when your site gets beyond a certain point, you may want to speed up the conversion process by having Pelican only read modified content. I haven’t tried this yet.

Once you’ve run one of those two publish commands, you can upload your content to wherever they will be hosted. If you’re using GitHub Pages like me, you’ll want to create a repository named, USERNAME.github.io, with your username. Any repo following this naming scheme will be recognized and served automatically. You can then simply push your content to the master branch of that repo, and wait a few minutes to see it appear at http://USERNAME.github.io!

Note, if you set up a CNAME earlier, you’ll also want to go to your domain name registrar and edit the DNS settings to point to the right place.

Import old blog

Since I was migrating posts out of Blogger, I did this in a three-step process. First, I used Blogger’s export XML functionality to retrieve my content. Second, I ran the following command to convert the XML to Markdown.

pelican-import -m markdown --feed --dir-page blog___.xml --dir-cat

Third, I wrote a python script to do a bit of cleanup. Haha That’s the hand-wavy magic answer. But you can see my script here and edit it to suit your needs.

Next steps

As I mentioned, Pelican is easily extended and customized. Check out pelican plugins for a large collection of extensions. So far, I’m only using tag_cloud. I also recommend going through all the available settings to see if you might like some non-default behavior. The particular theme that you choose may have other settings as well. Finally, you can edit style to your heart’s content. Or not at all.

If anything didn’t work quite right or if you’d like me to expand on any section, please email me or tweet at me. Happy Blogging!
